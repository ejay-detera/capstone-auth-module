# Subsystem Integration Guide

This guide details how to integrate a new subsystem (e.g., HRMS, Inventory Management, etc.) with the Capstone Authentication Module.

The Auth Module is designed as a centralized identity provider using a decoupled, token-based architecture. Any external service or subsystem can authenticate its users and enforce role-based access control by verifying tokens against this centralized service.

## 1. Network Integration

All services in the Capstone ecosystem communicate over a shared Docker network called `shared-capstone-network`. 

To allow your new subsystem to communicate securely with the Auth Module without traversing the public internet, ensure your subsystem's `docker-compose.yml` attaches to this network:

```yaml
# In your subsystem's docker-compose.yml
networks:
  shared-capstone-network:
    external: true

services:
  your_new_service:
    # ... your service config
    networks:
      - shared-capstone-network
```

## 2. API Gateway Routing (Nginx)

The ecosystem uses a centralized Nginx reverse proxy to route public traffic to the correct internal containers. You must update the `nginx.conf` in the Auth Module to expose your new subsystem's API and frontend.

Open `nginx/nginx.conf` and add new `location` blocks for your service:

```nginx
    # New Subsystem API Gateway
    location /api/new-subsystem/ {
        proxy_pass http://new_subsystem_api:8000/api/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header Content-Type $content_type;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_pass_request_body on;
    }

    # New Subsystem Frontend
    location /new-subsystem/ {
        set $upstream_new http://new_subsystem_web:5003;
        proxy_pass $upstream_new;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
    }
```

## 3. Registering Subsystem Permissions

The Auth Module handles role-based access control (RBAC). Permissions are scoped to specific subsystems via the `system` column in the database.

To register permissions for your new subsystem, add them to `auth-service/database/seeders/RolePermissionSeeder.php`:

```php
$permissions = [
    // ... existing permissions

    // New Subsystem Permissions
    ['name' => 'View Dashboard', 'slug' => 'new-subsystem.dashboard.view', 'system' => 'new-subsystem'],
    ['name' => 'Manage Records', 'slug' => 'new-subsystem.records.manage', 'system' => 'new-subsystem'],
];
```

You can then assign these permissions to existing or new roles within the seeder, and they will be manageable via the Auth Module's web interface.

## 4. Verifying Tokens in Your Subsystem

When a client makes a request to your subsystem, they will provide a Bearer token (generated by the Auth Module) in the `Authorization` header.

Your subsystem **must not** trust this token blindly. Instead, it should make a backend-to-backend internal HTTP request to the Auth Module's `/api/auth/verify-token` endpoint to validate the token and retrieve the user's details and permissions.

### Verification Endpoint Reference

**POST** `http://auth-service:8000/api/auth/verify-token` (Internal URL)

**Request Body:**
```json
{
    "token": "1|abcdef123456..."
}
```

**Success Response (200 OK):**
```json
{
    "valid": true,
    "user": {
        "id": 1,
        "email": "user@example.com",
        "first_name": "John",
        "last_name": "Doe",
        "role": "Manager",
        "department": "Sales",
        "permissions": [
            "new-subsystem.dashboard.view",
            "new-subsystem.records.manage"
        ]
    }
}
```

**Failure Response (401 Unauthorized):**
```json
{
    "valid": false,
    "message": "Invalid or expired token."
}
```

### Integration Example (Laravel Middleware in Subsystem)

If your new subsystem is built in Laravel, you can create a middleware to verify the token:

```php
<?php

namespace App\Http\Middleware;

use Closure;
use Illuminate\Http\Request;
use Illuminate\Support\Facades\Http;

class VerifyAuthModuleToken
{
    public function handle(Request $request, Closure $next)
    {
        $token = $request->bearerToken();

        if (!$token) {
            return response()->json(['message' => 'Unauthenticated.'], 401);
        }

        // Call the central Auth Module
        $response = Http::timeout(3)->post('http://auth-service:8000/api/auth/verify-token', [
            'token' => $token
        ]);

        if ($response->failed() || !$response->json('valid')) {
            return response()->json(['message' => 'Invalid or expired token.'], 401);
        }

        // Attach user data to the request for controllers to use
        $request->merge(['auth_user' => $response->json('user')]);

        return $next($request);
    }
}
```

## 5. Fetching Subsystem-Specific Permissions

If your subsystem's frontend needs to know which UI elements to show/hide based on permissions, it can hit the Auth Module's permissions endpoint, filtering by the system name:

**GET** `/api/auth/permissions?system=new-subsystem` (Requires Bearer token)

This will return only the permissions assigned to the logged-in user that belong to `new-subsystem`.

## 6. Audit Logging (Optional)

Currently, the Auth Module pushes specific login/logout audit events to the CRMS Vendor Management service. If your new subsystem also requires centralized audit logs from the Auth Module, consider refactoring the hardcoded HTTP pushes in `AuthController.php` (e.g., `pushToCrmsAuditLog`) into an event-driven architecture (like RabbitMQ or Redis Pub/Sub). This will allow any subsystem to listen to authentication events asynchronously.
